# Pricing & Risk Workstation — нюансы, проблемы, вопросы и план устранения

**Дата:** 2026-07-09.
**Контекст:** итоги сессий постройки Pricing Workstation (37 продуктов × 87 движков), Desk Risk (ladder + сценарии) и Market Risk (HypPL / VaR / ES / backtesting) по инструкции `calypso_risk_modules_models_detailed.md`. Коммиты `6b60290…efef0d7`.

Формат каждого пункта: **описание → статус → вопрос к тебе (если есть) → предложение → план**.

## 1. Данные (маркет дата)

### 1.1. История FX-фиксингов отсутствовала — валютный риск был скрыт ✅ исправлено
В сторе было всего 12 снапшотов `fx_rates` (по одному на ингест). FX-фактор в HypPL/VaR был нулевым: VaR99 демо-книги показывал 790 тыс., а после подключения фактора — **2.12 млн**. Валютный форвард на $1 млн нес скрытый риск, которого «не видел» весь Market Risk.
- **Сделано:** шаг `cbr_fx` в BackfillJob (дневные официальные фиксинги USD/EUR/CNY через `XML_dynamic.asp`), бэкфилл 5 лет (1247 точек/пара), фактор `USDRUB:fix` подключён в `factor_shifts`.
- **Остаток проблемы:** фиксинги ЦБ публикуются «на завтра» и не каждый торговый день — из 500 дней только 284 с ненулевым движением (на пропусках carry). Годовая вола получается ~10.9% — вероятно, занижена против реальной динамики.
- **Вопрос:** какой ряд предпочесть для FX-фактора — фиксинги ЦБ (как сейчас) или ряд из фьючерсов Si/CR (точнее дневная динамика, есть в price_history)? Нужны ли отдельные факторы EUR/RUB и CNY/RUB (истории уже загружены)?
- **План:** нормализовать конвенцию дат фиксинга (сдвиг на день), добавить фьючерсный ряд как альтернативный фактор, EUR/CNY факторы — при появлении соответствующих позиций.

### 1.2. Нет рыночных волатильностей ставок (IRVOL) — swaption cube на демо-котировках
Cap/floor, свопционы, CMS ценятся с ручной волой; `swaption_cube` калибруется по демо-матрице (в реестре так и помечено: «no market IRVOL source yet»).
- **Вопрос:** есть ли доступ к платному источнику (Cbonds, Bloomberg, НФА-квоты RUB swaptions)? Если нет — устроит ли model-implied подход (калибровка HW/G2++ к историческим переоценкам ОФЗ) как временная мера?
- **Предложение:** ручной ввод ATM-матрицы через UI (загрузка CSV с брокерских терминалов) + хранение в сторе с версионированием.
- **План:** экран Market Data → Volatility дополнить редактором swaption-матрицы; PricingService уже умеет `cube_id`.

### 1.3. Кредитные данные: CDS-рынка по RU именам нет — hazard-кривые демо
CDS, risky bond, CDO ценятся на `hazard_1t_demo` / `hazard_hy_demo`. Реальных CDS-спредов не будет (рынок закрыт с 2022).
- **Предложение (сильное):** калибровать hazard-кривые из **z-спредов корпоративных облигаций**, которые уже есть в сторе (CORP_T1/T2/T3 калибруются из TQCB) — риск-нейтральные PD из спредов при recovery-допущении. `curves/hazard.py` уже умеет бутстрап из z-спредов.
- **Вопрос:** какое recovery-допущение принять базовым (стандартные 40%? для RU корпоратов часто берут 20–30%)? Нужны ли per-issuer кривые или достаточно tier-уровня (T1/T2/T3)?
- **План:** шаг ингеста «hazard из корп-спредов» → кривые `HAZARD_T1/T2/T3` в снапшоте → choices в кредитных продуктах воркстейшена.

### 1.4. ATM IV только для FORTS-андерлаингов; акции получают RV30
Автозаполнение волы для Si/RTS/BR берёт IV с калиброванной SABR-поверхности, а для акций (SBER и т.п.) — только 30-дневную реализованную. Маппинга «акция → код опционного андерлаинга FORTS» (SBER→SR, GAZP→GZ) нет.
- **Предложение:** таблица маппинга ~20 ликвидных имён + фолбэк RV30 (как сейчас).
- **План:** словарь в `api/underlying.py`, IV с поверхности фьючерса на акцию.

### 1.5. КБД-история только 5 теноров — фактор ставок в VaR фактически параллельный
Бэкфилл пишет KBD:0.25/1/2/5/10Y, в HypPL используется только 5Y. Нет twist/steepener-факторов, нет bucketed rate VaR.
- **Предложение:** расширить сетку бэкфилла до полной (11+ теноров уже в снапшотах), в Market Risk перейти на 3 PCA-фактора кривой (level/slope/curvature) — `risk/historical_var.pca_var` уже реализован.
- **План:** фаза 5 (вместе с Incremental VaR): multi-tenor сдвиги → bucketed переоценка бондов/свопов.

### 1.6. Нет истории имплайд-волатильностей опционов
`option_quotes`/vol points живут в последнем снапшоте; исторических IV-рядов нет → vega-фактор в VaR — только RVI-прокси (индекс волатильности RTS).
- **Предложение:** начать копить ATM IV per underlying ежедневно в `time_series` (после каждого EOD-ингеста сохранять `IV:{code}:atm`).
- **План:** маленький шаг в EodIngestJob; через 3–6 месяцев появится собственная история для vega-VaR.

### 1.7. Экспирация фьючерсов не всегда заполняется
`expiry_T` берётся из `instrument_ref.last_trade_date`, но у части контрактов (и у агрегатных «Si»-сущностей) поле пусто — автозаполнение T молча пропускается.
- **План:** фолбэк-парсинг кода контракта (месячная буква + год: M6 = июнь 2026) — детерминированно для FORTS-тикеров.

### 1.8. CPI для линкеров — ручной CSV
`load_cpi_csv` — ручная загрузка; автоматического источника Росстата/ЦБ нет (публикуют таблицами).
- **Вопрос:** линкеры (ОФЗ-ИН) в приоритете? Если да — договоримся о процессе обновления CSV (раз в месяц) или парсить страницу ЦБ.

### 1.9. Дивиденды — только история выплат, нет форвардной дивидендной кривой
Для длинных опционов на акции q берётся как trailing yield — грубо для дальних экспираций.
- **План:** низкий приоритет; можно добавить простую модель «подтверждённые + экстраполяция» позже.

### 1.10. Real-time в прайсинг не подключён
Прайсеры читают EOD-снапшот; живые котировки (10s TTL) есть только в Market Data UI.
- **Вопрос:** нужен ли intraday-режим прайсинга (спот из live-котировки вместо EOD)? Это просто для спота (галка «live spot» в underlying picker), сложнее для кривых/вол.

## 2. Методология / модели

### 2.1. Двойной сдвиг ставки в сценариях ✅ исправлено
В `scenarios_ws` rate-шок добавлялся и в поле `r`, и в `shift_bps`. При флэт-кривой (строится из `r`) шок удваивался: «Rate hike 2022» бил по IRS +800бп вместо +400бп. Исправлено + регрессионный тест (сценарий == ручной репрайс `r+400bp`).

### 2.2. Market Risk считается по демо-портфелю
Книга — 4 хардкод-позиции в `api/context.py` (колл, бонд 5Y, IRS 5Y, FX-форвард). Все VaR-цифры — про неё.
- **Вопрос (ключевой для приоритизации):** что за реальный портфель предполагается вести? Состав по классам активов? Это определит порядок фаз 4–6.
- **План:** фаза 6 — trade capture из воркстейшена («В портфель»), персистентное хранение, затем Market Risk по реальной книге. Предлагаю поднять её ПЕРЕД фазами 4–5.

### 2.3. Факторная карта полной переоценки грубая (4 фактора на всю книгу)
`full_reprice_pnl` применяет dS ко всем спотам, dvol ко всем волам, dr ко всем ставкам — независимо от имени/валюты/тенора. Для многоимённого портфеля per-name риск не различается (у SBER и GAZP один фактор IMOEX).
- **Предложение:** факторная модель уровня 2: per-name equity-факторы (история 30 имён уже в сторе), КБД по тенорам (см. 1.5), FX по парам.
- **План:** фаза 5; `Position` уже несёт `params` — нужен маппинг позиция→факторы.

### 2.4. RVI как единственный vol-фактор
RVI — индекс волатильности RTS; применять его сдвиги к волам всех классов активов — прокси.
- **План:** после накопления собственной IV-истории (1.6) — per-underlying vega-факторы.

### 2.5. Горизонт VaR — sqrt-time, а не overlapping windows
10d VaR масштабируется √10. В `risk/var.py` есть честные overlapping h-day окна (Basel-style) для return-рядов — в marketrisk не подключены.
- **План:** переключить horizon>1 на overlapping-окна HypPL (данных хватает: 1250 дней).

### 2.6. Basel traffic light обобщён
Пороговые 5/10 пробоев определены для 99%/250д; я обобщил как 2×/4× ожидаемых пробоев для любого уровня доверия. Нестандартно, но осознанно (помечено в коде). При 99%/250д совпадает с Basel точно.

### 2.7. Kupiec отклоняет и «слишком консервативную» модель
0 пробоев при ожидаемых 2.5 → reject (p=0.025). Это правильно статистически, но в UI стоит пояснить, что reject ≠ «модель занижает риск». Сейчас подпись есть, но без направления смещения.
- **План:** добавить направление (консервативна/агрессивна) в backtest-карту.

### 2.8. Variance swap — синтетический стрип
Fair strike реплицируется из параметрического смайла (флэт/skew), не из реальных котировок опционов.
- **Предложение:** для Si/RTS строить стрип из реальных FORTS-цепочек (settle-цены уже в сторе).
- **План:** движок «market strip» вторым в продукте variance_swap.

### 2.9. Корреляции мульти-ассет продуктов — ручные
Spread/basket/rainbow берут попарную ρ одним числом. При этом basket_note уже умеет эмпирические корреляции из стора.
- **План:** кнопка «корреляции из истории» для basket/rainbow (переиспользовать код basket_note).

### 2.10. Статусы моделей: почти всё Approximation, часть Prototype
Prototype (слабые): `mc_lsm`, `asian` (арифметик MC без строгой валидации), `multi_asset`, `structured_autocall`, `structured_basket_note`, `frn` (нет par-reset логики!), `callable_bond` (BDT, флэт-вола). UI честно показывает статус, но продавать цифры из Prototype нельзя.
- **Вопрос:** запускать программу валидации (промоушен Approximation→Validated по списку `validation_candidates()`)? Это отдельный пласт работы (бенчмарки против внешних референсов).
- **План:** минимум — заменить FRN-движок (проекция форвардов уже есть в IRS-коде), довалидировать asian против Turnbull-Wakeman.

### 2.11. MC-шум в ladder/сценариях
Для MC-движков (rough Bergomi, TARN, автоколл) ladder на 15+ точках шумит; seed фиксирован (детерминированно), но профиль может «пилить».
- **План:** common random numbers между точками ladder (пробросить seed) + предупреждение в UI при MC-движке.

## 3. Архитектура / техника

### 3.1. Три параллельных каталога прайсинга
`/catalogue` (легаси, 10 vanilla), `/instruments/bond` (14 бондов), `/pricing/catalogue` (37 универсальных). Легаси Options-пейн из UI убран, но код (`PricingView`) и эндпоинт остались.
- **Вопрос:** удалить легаси `/catalogue` + PricingView? (BondPane трогать не предлагаю — он глубже воркстейшена по FI-аналитике: KRD, кэшфлоу-таблицы, curve-панель.)
- **План:** после твоего ок — удаление + чистка `ParamFieldView`/`ServerDownView` в общий файл.

### 3.2. Shared mutable state в мосте
`_svc.market_data = CONTEXT.market` присваивается в каждом pricing-хендлере — потокобезопасно только потому, что присваивается одно и то же. Грязный паттерн.
- **План:** передавать market_data параметром или конструировать PricingService с CONTEXT.market при старте (после стабилизации CONTEXT.reload).

### 3.3. Кэш HypPL не знает о составе портфеля
Ключ кэша — (snapshot_id, window). Когда появится trade capture (фаза 6), изменение книги обязано сбрасывать кэш.
- **План:** добавить хэш состава портфеля в ключ — заложить при фазе 6.

### 3.4. Мост не перезапускается при обновлении кода
Старые процессы висят на 8765 (в этой сессии дважды ловил 404 от старого сервера).
- **Предложение:** `scripts/restart_bridge.sh` (kill+start+health-check) или launchd-агент с KeepAlive.
- **План:** скрипт — 15 минут работы, сделаю в следующую сессию, если не возражаешь.

### 3.5. SwiftPM инкрементальный линкер иногда ломается
После больших правок линкер падал на несвязанных типах (`MDLiveQuote`); лечится `rm -rf .build`. Известная болячка SwiftPM — просто фиксирую как нюанс.

### 3.6. Клиент не применяет min/max из ParamSpec
Сервер шлёт границы, Swift-поля их игнорируют; ошибки уходят на сервер (422/400). Флаг `advanced` тоже не используется — все параметры развёрнуты.
- **План:** клиентская валидация + сворачивание model/numerical групп по флагу advanced (UI-полировка, низкий приоритет).

### 3.7. Лимиты сессий субагентов
Gap-analysis агент умер по session limit («resets 11:10pm»). Работал дальше инлайн — на результат не повлияло, но большие параллельные воркфлоу под вечер рискованны. Оперативный нюанс, не проектный.
